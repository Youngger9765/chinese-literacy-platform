## 📋 文件摘要

本文件基於已完成的市場研究、技術選型、教學理論，以產品經理專業角度，規劃朗朗上口（Chinese Literacy Platform）的執行策略。

**核心理念**：基於已驗證的成功經驗（朗朗上口 + Duotopia），採用「拿來主義」快速建置，解決台灣國小學生聽說讀寫四項能力的真實教學痛點。

### 1.3 如何逐步解開迷霧 🔍

#### 驗證策略總覽

**三階段驗證循環**：
```
Phase 1（Week 1-8）：技術驗證
    ↓
Phase 2（Week 9-16）：產品驗證
    ↓
Phase 3（Week 17-20）：商業驗證
    ↓
Phase 4（Week 21-24）：規模化驗證
```

#### Phase 1：技術驗證（Week 1-8）

**關鍵問題**：技術可行嗎？

**驗證項目**：
- [ ] **Week 1**：Azure Speech SDK zh-TW 準確率測試（目標 > 90%）
- [ ] **Week 2**：朗朗上口架構移植可行性（校班生師課資料模型）
- [ ] **Week 8**：語音分析 + 即時回饋完整流程（端到端測試）

**成功標準**：
- Azure STT 準確率 > 90%
- 朗讀分析回應時間 < 5 秒
- 曾教授確認教學邏輯正確

**失敗應對**：
- 若 Azure < 90%，立即切換 Google Speech-to-Text
- 若回應時間 > 5 秒，優化演算法或增加伺服器資源

#### Phase 2：產品驗證（Week 9-16）

**關鍵問題**：使用者喜歡嗎？

**驗證項目**：
- [ ] **Week 14**：內部測試（5 位教師、20 位學生）
- [ ] **Week 16**：收集回饋（NPS、使用時長、功能偏好）

**成功標準**：
- 教師 NPS ≥ 40（推薦者比例 > 批評者）
- 學生持續使用率 > 70%（第 4 週仍在用）
- 教師認為系統減輕教學負擔（問卷 ≥ 4.0/5.0）

**失敗應對**：
- NPS < 40：訪談使用者，找出核心痛點，快速迭代
- 使用率 < 70%：檢視卡點（介面、流程、激勵機制），調整 UI/UX

#### Phase 3：商業驗證（Week 17-20）

**關鍵問題**：能賺錢嗎？

**驗證項目**：
- [ ] **Week 18**：成本報告（Azure + LLM + Storage 實際用量）
- [ ] **Week 20**：付費意願調查（教師、家長、學校）

**成功標準**：
- 每位學生月均成本 < $10（Azure + LLM + Storage）
- 付費意願 ≥ 30%（願意支付訂閱費）
- 明確商業模式（學校訂閱 vs 家長自費）

**失敗應對**：
- 成本 > $10：優化用量（快取、批次處理、降低 LLM 呼叫頻率）
- 付費意願 < 30%：調整商業模式（免費增值、政府補助、企業贊助）

#### Phase 4：規模化驗證（Week 21-24）

**關鍵問題**：能規模化嗎？

**驗證項目**：
- [ ] **Week 22**：Beta 測試（2-3 所學校、≥ 50 位學生）
- [ ] **Week 24**：規模化挑戰盤點（技術、支援、訓練）

**成功標準**：
- 系統穩定度 > 99%（30 位學生同時使用）
- 教師訓練時間 < 30 分鐘（自助上手）
- 客服需求 < 5%（學生自助解決問題）

**失敗應對**：
- 穩定度 < 99%：加強監控、錯誤處理、備援機制
- 訓練時間 > 30 分鐘：簡化流程、製作教學影片、到校服務

## 第二章：執行策略（三步走）

### 2.1 Phase 1：驗證與準備（Week 1-2）

#### 目標（Objectives）

**O1**：驗證技術可行性
- Azure Speech SDK zh-TW 準確率 > 90%

**O2**：建立核心架構
- 校班生師課資料模型完成
- 教師端基礎管理介面可用

**O3**：確認教學理論對齊
- 與曾教授確認功能設計符合閱讀科學

#### 關鍵成果（Key Results）

**KR1**：技術驗證完成
- [ ] Azure STT 測試報告（10 段朗讀，計算 WER）
- [ ] 準確率 ≥ 90% 或確認備選方案（Google STT）

**KR2**：架構建立完成
- [ ] 資料庫 Schema 設計完成（School、Class、Teacher、Student、Course）
- [ ] 教師端可建立學校、班級、學生
- [ ] 學生端可登入並查看課程列表

**KR3**：教學驗證完成
- [ ] 與曾教授 Demo #1（展示架構規劃）
- [ ] 曾教授確認系統架構符合教學需求

#### 待驗證假設（Assumptions to Validate）

**A1**：Azure Speech SDK 支援國小學生朗讀
- 假設：國小學生發音（不標準、方言腔調）仍可達 90% 準確率
- 驗證方式：錄製 10 段國小低年級學生朗讀，測試辨識結果

**A2**：校班生師課架構適用於國語文教學
- 假設：Duotopia 的多層級架構可直接應用於國語文情境
- 驗證方式：與曾教授討論學校實務需求，確認資料模型設計

**A3**：教師願意手動上傳課文
- 假設：教師接受「貼上課文」的操作方式（而非整合教科書 API）
- 驗證方式：Demo 給潛在測試教師，詢問操作難度與意願

#### 風險與應對（Risks & Mitigation）

**R1**：Azure STT 準確率不足（機率：中，影響：高）
- 應對：Week 1 立即測試，若 < 90% 改用 Google Speech-to-Text
- 備案：Google STT 支援 zh-TW，準確率文獻記載 > 90%

**R2**：學生開發進度落後（機率：中，影響：中）
- 應對：Young 每週 Code Review，及時發現問題，調整任務分配
- 備案：Phase 1 砍次要功能（如：CSV 匯入，改為手動新增）

**R3**：曾教授時間無法配合（機率：低，影響：中）
- 應對：提前確認曾教授 Demo 時間，彈性調整 Demo 日期
- 備案：由方大哥代為確認教學邏輯，事後補充曾教授意見

#### 目標（Objectives）

**O1**：完成核心學習循環

**O2**：教師可追蹤學習進度
- 班級統計報告、個別學生表現、卡點預警

**O3**：驗證產品價值
- 內部測試（5 位教師、20 位學生）

#### 關鍵成果（Key Results）

**KR1**：朗讀分析模組完成（Week 7-8）
- [ ] Azure STT 整合完成
- [ ] 語速計算準確率 ± 5%
- [ ] 錯誤分類（跳字、加字、讀錯）準確率 > 85%
- [ ] 分析回應時間 < 5 秒
- [ ] 曾教授 Demo #2 驗證教學邏輯

**KR2**：即時回饋系統完成（Week 9-10）
- [ ] 六階段回饋流程完整（朗讀結果、錄音播放、逐句對比、錯字詞清單、練習建議、AI 分析）
- [ ] AI 分析建議有教學價值（曾教授認可）

**KR3**：生字學習模組完成（Week 13-14）
- [ ] 字型、字音、字義、筆順四模組完整
- [ ] 部件拆解顯示正確
- [ ] 筆順動畫流暢
- [ ] 發音練習可評分
- [ ] 曾教授 Demo #3 驗證教學邏輯

**KR4**：教師儀表板完成（Week 15-16）
- [ ] 班級統計報告可視化
- [ ] 個別學生表現追蹤
- [ ] 卡點預警通知
- [ ] 報告可匯出（PDF / Excel）

**KR5**：內部測試完成（Week 14-16）
- [ ] 5 位教師使用系統 2 週
- [ ] 20 位學生完成 ≥ 3 次練習
- [ ] 收集回饋問卷（NPS、使用時長、功能偏好）
- [ ] 產出改進清單

#### 待驗證假設（Assumptions to Validate）

**A1**：即時回饋真能提升學習效果
- 假設：學生看到錯誤後會主動改進（vs 只是看過就忘）
- 驗證方式：追蹤學生 10 次練習的正確率趨勢，檢驗是否提升 ≥ 10%

**A2**：生字學習模組對錯字有幫助
- 假設：學生進入生字學習後，該字再次錯誤率降低
- 驗證方式：比對「學習前錯誤率」vs「學習後錯誤率」

**A3**：教師真的減輕教學負擔
- 假設：教師不需逐一聽讀，節省時間
- 驗證方式：問卷詢問「使用系統後教學負擔是否減輕」（5 分量表）

**A4**：AI 分析建議有教學價值
- 假設：LLM 生成的建議符合教學邏輯（vs 流於表面）
- 驗證方式：曾教授審查 AI 建議內容，確認符合閱讀科學

#### 風險與應對（Risks & Mitigation）

**R1**：語音分析準確率不足（機率：中，影響：高）
- 應對：人工標註 100 段朗讀，計算 AI 與人工標註一致性
- 備案：若一致性 < 85%，調整文字對齊演算法（參考朗朗上口）

**R2**：學生覺得介面複雜（機率：中，影響：中）

## 第四章：教育知識整合

### 4.1 曾世杰教授教學法應用

#### 七大教學法系統實踐

**1. 部件教學法**

**教學原理**：
- 教導孩子拆解國字（如：「清」= 氵 + 青）
- 建立字感，預測陌生字能力

**系統實踐**：
- ✅ **生字學習模組**：顯示部件拆解（氵 + 青）
- ✅ **部件說明**：「氵（水部）與水有關，青是聲音」
- ✅ **相關字延伸**：「清、情、晴、請」都有「青」聲符

**驗收標準**：
- [ ] 每個生字都必須顯示部件拆解
- [ ] 部件說明包含「形符」（意義）與「聲符」（讀音）
- [ ] 提供至少 3 個相關字延伸

---

**2. 聲韻覺識**

**教學原理**：
- 透過「玩聲音」（拼音、拆音、換音）強化聲音敏感度

**系統實踐**：
- ✅ **注音符號顯示切換**：Space 鍵隨時開關
- ✅ **拼讀練習**：「清」→ 「ㄑㄧㄥ」（生字學習模組）
- ⚠️ **換音遊戲**：Phase 4+ 考慮（非 MVP 必要）

**驗收標準**：
- [ ] 注音符號可隨時開關（Space 快捷鍵）
- [ ] 懸浮顯示於文字上方（不影響閱讀）
- [ ] 生字學習模組包含拼讀練習

---

**3. 流暢朗讀**

**教學原理**：
- 識字自動化（不需一字一頓），大腦才有餘裕處理文意
- 流暢度 = 速度 × 準確度（兩者並重）

**系統實踐**：
- ✅ **語速目標**：150-180 字/分（識字自動化標準）
- ✅ **正確率目標**：> 90%（準確度不能犧牲）
- ✅ **達標判定**：兩者都必須達標才顯示 ✅
- ✅ **個別化目標**：教師可為每位學生調整目標

**驗收標準**：
- [ ] 系統同時追蹤語速與正確率
- [ ] 兩者都必須達標才顯示 ✅
- [ ] 達標標準可由教師個別調整

---

**4. 針對性練習**

**教學原理**：
- 避免無效罰抄，只練錯的字
- 多感官學習（聽、看、寫）

**系統實踐**：
- ✅ **錯字詞清單**：自動彙整所有讀錯的字
- ✅ **發音按鈕**：點擊 🔊 聽正確唸法
- ✅ **部件教學**：顯示部件拆解
- ✅ **多感官學習**：聽（發音）、看（部件）、寫（筆順）

**驗收標準**：
- [ ] 所有讀錯的字自動加入清單
- [ ] 每個錯字提供發音按鈕
- [ ] 每個錯字顯示部件拆解
- [ ] 可點擊進入完整生字學習模組

---

**5. 早期介入**

**教學原理**：
- 發現落後即刻介入，不需等待醫學鑑定
- 一、二年級是黃金期

**系統實踐**：
- ✅ **學習預警通知**：
  - 超過 7 天沒練習 → 通知教師
  - 連續 3 次未達標 → 通知教師
  - 同一生字錯誤 ≥ 3 次 → 卡點提醒
- ✅ **提醒內容具體**：「小明在『清』這個字錯了 3 次，建議進行部件教學」

**驗收標準**：
- [ ] 系統每日自動檢查學生學習狀態
- [ ] 符合條件立即通知教師
- [ ] 通知內容包含具體行動建議
- [ ] 教師可設定通知頻率

---

**6. 後設認知**

**教學原理**：
- 培養自我監控、自我調整能力
- 「我知道我哪裡不會」是學習的關鍵

**系統實踐**：
- ✅ **重聽錄音**：學生聽到自己的聲音，覺察「我讀錯了」
- ✅ **逐句對比**：原文 vs 實讀，精準定位錯誤
- ✅ **學習曲線圖**：看到進步軌跡
- ✅ **錯字詞清單**：建立「錯誤清單」習慣

**驗收標準**：

#### 五大支柱整合

**美國閱讀科學（Science of Reading）核心五大支柱**：

1. **Phonemic Awareness（聲韻覺識）**
2. **Phonics（自然發音）**
3. **Fluency（流暢度）**
4. **Vocabulary（詞彙）**
5. **Comprehension（理解）**

**系統對應**：

| 支柱 | 英文閱讀 | 中文閱讀 | 系統功能 |
|------|---------|---------|---------|
| **Phonemic Awareness** | 音素覺識 | 聲韻覺識（注音符號） | 注音符號顯示切換、拼讀練習 |
| **Phonics** | 自然發音 | 部件教學（形聲字） | 部件拆解、聲符說明 |
| **Fluency** | 流暢朗讀 | 流暢朗讀 | 語速 + 正確率雙重評估 |
| **Vocabulary** | 詞彙學習 | 生字學習 | 字型、字音、字義、筆順 |
| **Comprehension** | 閱讀理解 | 閱讀理解 | 蘇格拉底對話系統 |

**Simple View of Reading**：
```
閱讀能力 = 解碼能力 × 語言理解
```

**系統實踐**：
- ✅ **解碼能力**：朗讀分析（流暢度）+ 生字學習（識字自動化）
- ✅ **語言理解**：蘇格拉底對話（理解力評估）
- ✅ **兩者並重**：不能只練朗讀而忽略理解

### 6.1 Web vs. App 策略

#### 技術選擇決策

**決策**：Web 優先（MVP）

**理由**：

| 面向 | Web 瀏覽器 | 原生 App |
|------|-----------|----------|
| **開發成本** | 低（一套程式碼） | 高（iOS + Android 兩套） |
| **學習曲線** | 低（團隊熟悉 Web） | 高（需學習 Flutter / React Native） |
| **維護成本** | 低（改一次全平台更新） | 高（兩套程式碼分別維護） |
| **部署速度** | 快（即時更新） | 慢（App Store 審核） |
| **跨平台** | ✅ 電腦、平板、手機瀏覽器 | ⚠️ 需分別開發 iOS、Android |
| **離線功能** | 有限（PWA 可支援） | ✅ 完整支援 |
| **效能** | 中（瀏覽器限制） | 高（原生效能） |
| **App Store** | 不需要 | 需上架審核 |

**結論**：
- ✅ **MVP（Phase 1-3）**：Web 瀏覽器（響應式設計）
- ✅ **Beta 測試（Phase 4）**：評估需求，決定是否需要 PWA
- ⚠️ **正式上線後**：若需求強烈，Phase 5 使用 Capacitor 轉原生

---

#### 未來擴展路徑

**Phase 4**：PWA（Progressive Web App）
- 保留 Web 程式碼
- 增加離線功能（Service Worker）
- Android 可安裝（iOS 受限）
- 開發成本：+ 2 週

**Phase 5**：Capacitor（Web to Native）
- 保留 Web 程式碼
- 轉為原生 App（iOS + Android）
- 可上架 App Store
- 開發成本：+ 4 週

**Phase 6+**：Flutter（完全重寫）
- 效能最佳
- 需重寫全部程式碼
- 開發成本：+ 3 個月
- **僅在必要時考慮**（如：需要複雜動畫、遊戲化）

## 第七章：決策矩陣

### 7.1 技術選型決策

#### 決策 1：後端框架選擇

**問題**：FastAPI (Python) vs Node.js + Express？

**選項**：

| 選項 | 優勢 | 劣勢 | 適用場景 |
|------|------|------|---------|
| **FastAPI (Python)** | ✅ AI/ML 生態豐富<br>✅ Azure SDK 官方支援佳<br>✅ 語音分析演算法易實作 | ⚠️ 並發效能略低<br>⚠️ 團隊需學習 Python | 需大量 AI/ML 處理 |
| **Node.js + Express** | ✅ 並發效能高<br>✅ 團隊熟悉（JavaScript）<br>✅ 生態成熟 | ⚠️ AI/ML 工具鏈不如 Python<br>⚠️ Azure SDK 支援較弱 | 需高並發、實時通訊 |

**建議**：**FastAPI (Python)**

**理由**：
1. 語音分析演算法需 Python（文字對齊、錯誤分類）
2. Azure Speech SDK Python 版本功能最完整
3. 未來可能需 ML 模型（如：手寫辨識、筆順評估）
4. 並發需求不高（MVP 支援 30 人同時使用即可）
5. Young 熟悉 Python，可指導團隊

**風險應對**：
- 若團隊 Python 學習曲線過陡，改用 Node.js（犧牲部分 AI/ML 功能）

---

#### 決策 2：資料庫選擇

**問題**：PostgreSQL vs MySQL vs MongoDB？

**選項**：

| 選項 | 優勢 | 劣勢 | 適用場景 |
|------|------|------|---------|
| **PostgreSQL** | ✅ 功能強大（JSON、全文搜尋）<br>✅ 資料一致性高<br>✅ 適合複雜查詢 | ⚠️ 學習曲線略高 | 需複雜查詢、資料一致性 |
| **MySQL** | ✅ 生態成熟<br>✅ 學習曲線低 | ⚠️ 功能較弱<br>⚠️ JSON 支援不佳 | 簡單 CRUD 應用 |
| **MongoDB** | ✅ 彈性 Schema<br>✅ 水平擴展容易 | ⚠️ 資料一致性弱<br>⚠️ 不適合複雜關聯查詢 | 需彈性 Schema、大量寫入 |

**建議**：**PostgreSQL**

**理由**：
1. 校班生師課多層級關聯，需複雜 JOIN 查詢
2. JSON 支援（錯誤詳情、AI 分析結果可存 JSONB）
3. 全文搜尋（未來可能需搜尋課文內容）
4. Duotopia 使用 PostgreSQL，經驗可直接移植

---

#### 決策 3：語音識別服務

**問題**：Azure Speech SDK vs Google Speech-to-Text vs OpenAI Whisper？

**選項**：

| 選項 | 優勢 | 劣勢 | zh-TW 支援 | 成本 |
|------|------|------|-----------|------|
| **Azure Speech SDK** | ✅ Pronunciation Assessment<br>✅ Duotopia 已驗證 | ⚠️ 準確率需實測 | ✅ 官方支援 | $1 / 60 分鐘 |
| **Google STT** | ✅ 準確率高（文獻 > 90%）<br>✅ 生態成熟 | ⚠️ 無 Pronunciation Assessment | ✅ 官方支援 | $1.44 / 60 分鐘 |
| **OpenAI Whisper** | ✅ 開源免費<br>✅ WER 10.3%（業界最佳） | ⚠️ 需自建伺服器<br>⚠️ GPU 成本高 | ✅ 支援 | 自建成本高 |

**建議**：**Azure Speech SDK（主）+ Google STT（備）**

**理由**：
1. Azure Pronunciation Assessment 可評估發音品質（停頓、語調）
2. Duotopia 已驗證 Azure 整合經驗
3. 成本較低（$1 vs $1.44）
4. 若 Week 1 測試準確率 < 90%，立即切換 Google STT

### 7.3 商業模式決策

#### 決策 6：定價策略

**問題**：免費 vs 訂閱 vs 買斷？

**選項**：

| 選項 | 優勢 | 劣勢 | 適用場景 |

  - 學校統一採購

  - 完整功能（蘇格拉底對話、進階報告）

**Phase 6+**：政府補助
- 申請教育部數位教學計畫
- 偏鄉學校免費使用

**理由**：
1. Beta 階段免費試用，降低學校參與門檻
2. 正式上線後採免費增值，擴大用戶基數

### B. 名詞解釋

**WER（Word Error Rate）**：
- 語音識別錯誤率，越低越好
- 計算公式：(替換 + 刪除 + 插入) / 總字數

**NPS（Net Promoter Score）**：
- 淨推薦值，衡量用戶推薦意願
- 計算公式：推薦者比例 - 批評者比例
- 分數範圍：-100 ~ +100
- 優秀標準：> 50

**MVP（Minimum Viable Product）**：
- 最小可行產品，包含核心功能，快速驗證產品價值

**PWA（Progressive Web App）**：
- 漸進式網頁應用，Web 加上離線功能，可安裝至手機

**Capacitor**：
- Web 轉原生工具，保留 Web 程式碼，生成 iOS/Android App
